# 다음 단계

이번 MVP 이후 우선순위대로 정리한 후속 작업이다. 현재 구현되지 않은 항목은 인터페이스만
준비되어 있거나(`LegalSourceProvider` 등) 아예 없는 상태이며, 아래 항목들을 가짜로
구현하지 않았다(§21 금지사항 준수).

## 국가법령정보 API 연동

- `src/lib/legal-sources/national-law-provider.ts`는 `LegalSourceProvider` 인터페이스의
  자리표시자로, 호출 시 항상 "설정되지 않음" 오류를 던진다.
- 실제 구현 전 확인 필요: 국가법령정보 공동활용 Open API의 신청 절차, 인증 방식(API 키/
  IP 화이트리스트), 응답 스키마(XML/JSON), 요청 제한(rate limit).
- 구현 시 `manual-upload-provider.ts`와 동일한 `LegalSourceProvider` 인터페이스를 만족하게
  작성하면 검색·수집 파이프라인 나머지 코드는 그대로 재사용 가능하다.

## 자치법규 자동 동기화

- 지자체별 자치법규정보시스템 API(있는 경우) 또는 국가법령정보 API의 자치법규 검색
  기능을 활용해 조례를 자동 수집.
- 자동 수집분과 관리자 수동 업로드분이 충돌할 때의 우선순위 정책 필요.

## 법령 개정 감지

- 주기적으로 외부 API를 폴링하거나 웹훅을 받아 현행 버전의 개정 여부를 감지.
- 개정 감지 시 관리자에게 알림을 주고, 자동으로 새 DocumentVersion 초안을 생성하되
  검토·승인 없이는 활성화하지 않는 워크플로 필요.

## 신구조문 비교

- 개정 전/후 DocumentVersion의 LegalChunk를 조문 번호 기준으로 매칭해 diff를 보여주는
  화면.
- `docs/DOCUMENT_IMPORT_GUIDE.md`의 "문서 버전 관리"에서 이미 버전 간 시행일 구분은
  지원하지만, 조문 단위 비교 뷰는 아직 없다.

## 법제처 법령해석 수집

- `OFFICIAL_INTERPRETATION` 문서 종류는 스키마·검색·위계 구분까지 준비되어 있으나, 법제처
  법령해석례를 자동으로 가져오는 기능은 없다(관리자 수동 업로드만 가능).

## 국토교통부 유권해석 관리

- 위와 동일하게 문서 종류·검색·위계는 준비되어 있으나 자동 수집은 없음. 유권해석 특유의
  메타데이터(질의 요지, 회신 요지 등)를 구조화하려면 `LegalDocument`/`DocumentVersion`
  스키마 확장이 필요할 수 있다.

## 판례 검색

- `COURT_CASE`/`ADMINISTRATIVE_APPEAL` 문서 종류는 준비되어 있으나, 대법원 종합법률정보
  등 외부 판례 API 연동은 없음. 현재는 관리자가 판례 요지를 문서로 업로드하는 방식만
  지원.

## 전문가 검토 완료 답변

- `ChatMessage.reviewStatus`(PENDING/REVIEWED/NEEDS_CORRECTION/APPROVED)와
  `/admin/reviews` 화면으로 검토 워크플로의 기초는 마련했다.
- 다음 단계: "APPROVED" 답변을 별도로 표시하거나, 검토 완료된 답변 패턴을 향후 유사
  질문에 우선 제시하는 기능.

## 조직별 자료 분리 (멀티테넌시)

- 현재 스키마는 단일 조직 전제로 설계되어 있다(모든 문서·대화가 전역 공유).
- 조직 단위 데이터 분리가 필요하면 `Organization` 모델 추가, 모든 주요 테이블에
  `organizationId` 추가, 검색·권한 로직에 조직 필터 반영이 필요하다.

## 답변 품질 평가 데이터셋

- `AnswerFeedback`(도움됨/부정확/근거부족/오래된자료/지역오류/기타)이 이미 쌓이고 있으므로,
  이를 정답 세트로 축적해 회귀 테스트나 파인튜닝/프롬프트 개선 평가에 활용할 수 있다.
- 현재는 관리자 대시보드에 집계 수치만 표시되며, 별도 평가 파이프라인은 없다.

## 그 외 알려진 후속 과제

- **OAuth/SSO 미지원**: 이메일/비밀번호 인증만 지원한다(`src/lib/auth/session.ts`,
  `src/app/api/auth/login/route.ts`). 사내 SSO 연동이 필요해지면 별도 프로바이더 추가가
  필요하다.
- **감사로그 조회 화면 없음**: `AuditLog`에 로그인 성공/실패/계정 잠금/비밀번호 변경/
  사용자 관리 이벤트 등이 기록되고 있지만(`src/lib/audit/auditLog.ts`), 이를 관리자가
  화면에서 조회하는 UI는 아직 없다(DB를 직접 조회해야 함).
- **파일 다운로드 UI 없음**: `FileStorageProvider`(`src/lib/storage/`)로 원본 업로드
  파일을 저장하지만, 이를 사용자/관리자 화면에서 다시 다운로드하는 API·버튼은 아직 없다
  (파싱된 본문은 DB의 `rawText`로 이미 조회 가능하므로 원본 파일 자체는 감사 목적의
  보관 용도다).
- **OCR**: 스캔 PDF 자동 텍스트화는 미지원.
- **HWP 직접 파싱**: 미지원, PDF/DOCX/TXT 변환 안내만 제공.
- **벡터 인덱스(ANN)**: 현재 pgvector 컬럼에 ivfflat/hnsw 인덱스를 만들지 않았다(데이터
  규모가 작아 순차 스캔으로 충분). 데이터가 많아지면 인덱스 추가를 검토한다.
- **다른 차원의 임베딩 모델**: DB가 `vector(1536)`으로 고정되어 있어, 다른 차원의 임베딩
  모델(Voyage 일부 모델 등)을 쓰려면 마이그레이션이 필요하다.
